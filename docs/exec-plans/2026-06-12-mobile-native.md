# Mobile Native Implementation ExecPlan

## Background And Product Intent

The current product is a native macOS SwiftUI video downloader with a shared Swift `MoongateCore`, plus an independent Windows C#/WPF implementation. The requested direction is not a desktop UI clone. The mobile product must deliver a native phone workflow: add or import media, inspect formats and subtitles, download, translate, export subtitle files or rendered videos, and keep queue state honest across foreground/background interruptions.

July asked for iOS and Android versions, with iOS following Apple HIG, Android following Material Design, iOS support for Apple Intelligence paths, API key configuration on both platforms, and background download/render behavior where the OS allows it.

On 2026-06-13 July provided an external master plan at `/tmp/codex-remote-attachments/019ebc9b-42fa-7ac2-b061-2691a7c7656c/FBF9D48D-E507-452F-AE04-66F141D4EE59/1-codex-plan-4E1BC0C4-9275-42DB-BDB1-48BA8C5C6D41.md`. This local ExecPlan is now the execution baseline that merges that plan with current repository evidence.

- `Sources/MoongateCore` owns shared contracts, yt-dlp wrapping, page sniffing, settings, subtitle parsing/cleaning/translation, and ffmpeg burn-in.
- `Sources/Moongate` owns the macOS SwiftUI app, queue UI, WebKit login, dependency setup, and desktop interaction model.
- `Sources/moongate-cli` is the command-line validation surface for `MoongateCore`.
- `windows/` is a separate C# implementation with broader tests for queue behavior, parsing, translator requests, dependency planning, and WPF UI. It is useful as behavioral reference, not as a mobile foundation.
- `Sources/MoongateMobileCore/MobileModels.swift` and related tests are the current M1 mobile-contract starting point.
- There is now a verified unsigned local Xcode iOS app-bundle build for both simulator and device SDKs through `ios/MoongateiOSApp.xcodeproj` and `Scripts/build-ios-xcode.sh`, plus the existing SwiftPM `MoongateiOS` review shell and `MoongateiOSApp` host. Current mobile code also includes an iOS Keychain credential adapter, a pure mobile API-compatible translation provider contract, an Android `:core:domain` Kotlin contract module, and an Android `:app` Compose shell. The iOS/Android production entry points now start from empty live state; sample/mock state remains for previews and tests.
- The external master plan sets iOS as the first implementation target and Android as the second-stage target after the iOS product baseline is reviewable.

## Platform Evidence And Boundaries

- Xcode on this machine is `Xcode 26.5` with Swift `6.3.2`; the local iPhoneOS SDK is `iPhoneOS26.5.sdk`.
- The iPhoneOS SDK contains `FoundationModels.framework`. The Swift interface exposes `SystemLanguageModel.availability`, `isAvailable`, unavailable reasons (`deviceNotEligible`, `appleIntelligenceNotEnabled`, `modelNotReady`), `supportedLanguages`, `supportsLocale`, and `LanguageModelSession.respond/streamResponse`.
- The iPhoneOS SDK contains iOS 26 `BGContinuedProcessingTaskRequest`, including title/subtitle, queue/fail submission strategy, required resources, and optional GPU resource. The header states successful creation does not guarantee scheduler submission.
- `BGContinuedProcessingTask` must report progress and is subject to expiration based on system conditions and user input; stalled tasks may be expired.
- `URLSessionConfiguration.background(withIdentifier:)` exists for background transfers and can relaunch/resume apps when background session tasks complete or need authentication.
- There is no verified iOS 27 SDK or official iOS 27 API evidence in this environment. iOS 27 support must therefore be modeled as runtime capability detection and compatibility planning, not as a claim of implemented iOS 27-specific API support.
- Android implementation must use platform-native primitives rather than desktop process control. The current plan is WorkManager/foreground-service/user-initiated transfer architecture, scoped storage, Keystore-backed credentials, and Material 3 UI, to be verified against current official Android docs before code is written.
- Android documentation verification on 2026-06-13 confirmed the broad direction: persistent/background work should be modeled through WorkManager/foreground-service/user-initiated transfer style primitives with visible system mediation. This supports the existing boundary that Android background work must be represented as interruptible and resumable, not unlimited.

## Goals

- Build mobile-safe domain contracts in `MoongateMobileCore` that do not reference AppKit, UIKit, Android SDK, `Process`, Homebrew, POSIX signals, or desktop paths.
- Preserve desktop behavior while making mobile platform capabilities explicit.
- Create an iOS app shell using HIG-native structure: Add, Queue, Library, Settings.
- Add iOS secure settings and translation readiness: API key storage, OpenAI/Anthropic-compatible endpoints, Apple Translation/Foundation Models readiness and adapters.
- Add iOS download/export with background `URLSession` where suitable and continued-processing/background-task handling where suitable.
- Add Android native app using Material 3, Android secure storage, background work, local model/API translation readiness, and parity with the mobile core model.
- Build a repeatable verification matrix for unit tests, UI state tests, fixture media tests, accessibility checks, and real-device background behavior.
- Keep public mobile interfaces mockable before platform implementations exist: `MobileVideoCandidate`, `MobileVideoInfo`, `MobileFormatChoice`, `MobileSubtitleChoice`, `MobileDownloadRequest`, `MobileParser`, `MobileDownloadEngine`, `SubtitleProcessor`, `MobileTranslationProvider`, `RenderExporter`, `SecureCredentialStore`, and `TaskRepository`.

## Non-Goals

- Do not claim mobile parity with desktop yt-dlp site coverage until a mobile engine proves it.
- Do not bypass DRM, paywalls, login restrictions, platform access controls, or app-store review boundaries.
- Do not store API keys in ordinary JSON/UserDefaults/plain preferences/logs/crash text.
- Do not present background render/download as guaranteed when the OS can defer, expire, or kill it.
- Do not reuse the macOS dependency installer or Homebrew mental model on mobile.
- Do not add a broad service/backend architecture without explicit product and security confirmation.

## Architecture

### Shared Mobile Core

`MoongateMobileCore` should own pure mobile domain models and deterministic logic:

- mobile task state and progress snapshots;
- mobile error buckets and user-fixable/system-limited classification;
- mobile export profile and artifact/result models;
- platform capability matrix;
- background execution policy and interruption semantics;
- translation readiness and provider settings that exclude secret values.

### iOS

The iOS implementation should be a native SwiftUI target with:

- `TabView`/`NavigationStack` information architecture: Add, Queue, Library, Settings;
- Keychain-backed API key storage;
- `URLSession` background downloads for direct/media URLs that fit the platform model;
- `BGContinuedProcessingTaskRequest` on iOS 26+ for user-initiated continued processing, with progress reporting and expiration handling;
- `BGProcessingTaskRequest` only for deferrable work, not immediate user-visible rendering;
- Foundation Models and Apple Translation adapters behind runtime availability/readiness checks;
- explicit fallback states when a task needs foreground, power, network, storage, language download, or a supported device.

### Android

The Android implementation should be a native Kotlin/Compose app with:

- Material 3 `Scaffold`, NavigationBar, TopAppBar, bottom sheets/dialogs, and 48dp touch targets;
- Keystore/EncryptedSharedPreferences for API keys;
- WorkManager/foreground-service/user-initiated transfer strategy based on current Android docs;
- scoped storage and system document/photo export flows;
- a local model readiness/download/delete layer or an API-key translation mode;
- queue states mapped to Android notifications and resumable work.

## UX Acceptance Standards

- iOS and Android must both expose Add, Queue, Library, and Settings as top-level destinations. Queue must not exist only as a floating overlay or transient status hint.
- Add must be reviewable with mock services: input/import, analyzing, candidate selection, format/subtitle/translation selection, add-to-queue, failure, and retry.
- Queue must visibly distinguish active, waiting, paused, needs attention, done, failed, and cancelled states. Non-resumable work must not expose pause/resume actions.
- Every background task must state whether it is background eligible, foreground required, waiting for a system condition, expired, resumable, or non-resumable.
- Library must be independent of desktop paths and must track produced artifacts with share/export/delete and recovery paths for missing outputs.
- Mobile Settings must not contain Homebrew, desktop `Process`, Finder, or desktop cookie-export UI. API keys must be stored through secure platform storage and only surfaced as configured/missing state.
- Offline, permission denied, storage full, login restricted, model unavailable, language download needed, and background expiration states each need a distinct user-facing state and one primary action.
- iOS review must cover Dynamic Type, VoiceOver, Reduce Motion, and 44 pt touch targets. Android review must cover font scaling, TalkBack, edge-to-edge layout, 48 dp targets, and notification flows.

## Risks And Rollback

- Mobile site coverage may be much smaller than desktop unless a compliant engine or server-side option is approved.
- Background rendering may be deferred or expired; rollback is to foreground-only rendering with clear queue states.
- Apple Intelligence availability depends on device eligibility, user settings, locale/language support, model readiness, and OS version; fallback must be visible.
- App Store / Play review risk is real for video downloading, cookie/login handling, and third-party content saving.
- iOS API-key storage now has a Keychain adapter, reference-only model state, editable two-protocol cloud settings, provider/model UI, and a connection-test path that uses the API-compatible provider without leaking secrets. It also has unsigned local Xcode simulator/device SDK app-bundle build proof. It still needs simulator launch, signed device install, credential replacement/deletion UX beyond blank-key deletion, and real subtitle translation integration before it is shippable. Android now has a Keystore AES/GCM credential-store adapter and reference-only Settings state, but this is verified only by Swift static/boundary tests and source review; Android Gradle/APK/runtime validation remains blocked by missing local Gradle/wrapper.
- Android worker runtime must not be enabled until Gradle/JVM and device tests prove app/worker persistence interleavings. `JsonTaskRepository` now has source-level process/file locking and terminal-state merge protection, but this is still static/source evidence until real `:core:data:test`, WorkManager runtime, and device interleavings pass.
- Latest iOS/Android/UX read-only reviews keep mobile No-ship. iOS still lacks signed installable app proof, iOS-hosted test scheme/Keychain integration tests, entitlement/capability validation, and real background URLSession/BG continued-processing device QA. Android still lacks Gradle/APK/runtime proof, live WorkManager gates stay off, background resumability is not proven, the Add state machine is only partially covered at the domain/source boundary, and TalkBack/font-scale/runtime Material QA is open.
- If a mobile target destabilizes desktop builds, keep it in a separate target/project and preserve existing macOS/Windows paths.

- 2026-06-13: Treat iOS 26 APIs as locally verified by SDK; treat iOS 27 as unverified until an iOS 27 SDK/official docs are present.
- 2026-06-13: Background tasks must be modeled as best-effort and interruptible, never guaranteed.
- 2026-06-13: The external master plan is incorporated into this ExecPlan. iOS remains first; Android starts after iOS baseline review rather than in parallel with the initial iOS shell.
- 2026-06-13: M1 public contracts should be pure Swift/Foundation and mockable. Platform details such as Keychain, BGTask, Photos/Files, Keystore, WorkManager, notifications, HIG widgets, and Material components remain in platform layers.
- 2026-06-13: Ordinary JSON must not persist or restore direct source URLs on iOS, including plain HTTPS media URLs. Current-session in-memory source URLs may start a download, but relaunch recovery must fail clearly and require re-adding the source until an encrypted/Keychain-backed source store is designed.
- 2026-06-13: App-owned iOS task/config/transfer JSON and generated media/subtitle/render artifacts should be excluded from iCloud backup by default. This is privacy/storage hygiene only; it is not a substitute for Keychain-backed secrets or encrypted source recovery.
- 2026-06-13: Android WorkManager requests must continue to receive only an opaque `work_handle`. Restorable direct media request details may be handed to a future worker only through app-private `noBackupFilesDir/background-download-handoffs`, with strict HTTPS media URL rules and no URI userinfo, query, fragment, or credential-like source strings. Runtime enqueue remains gated until notification permission, foreground-service behavior, and the real worker body are verified.
- 2026-06-13: Android worker retry state must stay consistent with persisted task state. Retriable worker failures return a `Retrying` runtime result and keep the task in a non-final downloading snapshot; only the final non-retriable attempt may persist `FAILED`.
- 2026-06-13: Android task persistence must protect against stale app snapshots after worker completion. Source-level guardrails now include same-path JVM locking, file locking around repository read-modify-write, and a conservative terminal-state merge rule that keeps `COMPLETED` / `CANCELLED` from being overwritten by older non-terminal snapshots.
- 2026-06-13: Android worker handoff files must be atomic and per-handle locked before any WorkManager runtime gate can open. Source-level guardrails now write handoffs through a temporary file plus atomic move and wrap save/load/remove in same-handle JVM and file locks, while WorkRequest input remains an opaque `work_handle`.
- 2026-06-13: iOS background `URLSession` app-delegate completion handlers must be consumed only after queued delegate file-move and recovery-outcome recording work drains. Source/unit guardrails now require the delegate's finish-events path to mark sessions finished and defer handler consumption through a pending-event drain rather than releasing the handler immediately.
- 2026-06-13: Android direct-link entry points must parse as URIs before queueing and reject userinfo, query, and fragment before any foreground/background downloader path can see the source. Direct-link UI may stage only vetted HTTPS media-path URLs; generic webpages remain unsupported instead of fabricated.
- 2026-06-13: Android cloud translation readiness must not treat a saved API key as runnable by itself. Settings now models the two supported cloud protocols, base URL, model, credential reference, and transport-runtime availability separately; the UI may show the gaps, but it must not render a fake "test" or "enable translation" action until a real runtime adapter is wired and verified.
- 2026-06-13: Android Add domain state should use the shared `MobileAddSessionState` taxonomy. Source/static domain coverage can close idle/ready/unsupported/failed transitions, but the mobile product remains No-ship until Compose renders and tests the full Add UI state machine including analyzing and candidate-selection states.
