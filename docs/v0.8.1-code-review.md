# Moongate 0.8.1 Code Review + 歌词观感优化（以 YOASOBI「群青」为基准）

分支 `feature/language-first-subtitle-source`。本文是 0.8.1 的完整 code review，并记录针对"群青歌词观感差、时机不准"的优化与**实证结论**（含被否决的方向）。

---

## 0. 根因诊断（已用仓库内真实评测产物核验）

产物目录：`artifacts/subtitle_timing_eval/.../yoasobi-gunjou`。

1. **平台日文字幕实为乱码**：`source.vtt` 是 YouTube 自动字幕，含罗马音循环（`ni ni dare ni`、`anas`、`carano`）和错字（「一里/字買うん」）。质量门判 `garbledOrRepetitive` → **正确拒绝**，回退 Whisper（`source_candidates.json` 确认）。→ 源选定设计无误；**平台没有可用的人工时序源，Whisper 是唯一可用源**。
2. **Whisper 转写本身准**（「いつものように過ぎる日々に」「あくびが出る」均正确）；**中文用词也不差**（"随心所感地描绘""它也确在彼方"已具诗意）。
3. **"观感差" = 分段/cue 边界把句与词切碎**（源 `local-asr.ja.srt`）：
   - `感じ`(cue24) | `たまま`(cue25) — **「感じた」词中切断**，导致**误译**（感じ→"这种感觉"）。
   - `朝が`(cue3 悬空助词) | `降る`(cue4) — 「朝が降る」劈两条。
   - `手を伸ばせば` | `伸ばす` | `ほどに遠くへ` | `行く` — 一句剁成 4 条碎片。
   - 译文逐行锁死这些坏边界 → 每行像半句。
4. **LLM 重分段本应修复，却回退了**：生产 `local-asr.ja.zh-Hans.srt` 的时间戳/条数 = 源 43 条，说明 `resegmentForReadability` 整体回退到原始 cues。
5. **对照证明天花板可达**：`agent_llm_simulation/agent_translated.srt`（重分段版，29 条）把碎片合成完整诗句、修掉悬空行——接近字幕组水平。

> 注意：`viewing_quality/.../translated.srt`（43 条）由评测的**约束型翻译模拟器**生成（`agent_translation.prompt.md` 明确"不改编号/时间轴/不增删 cue"），**不等于**生产 `ConfiguredTranslator.translate` 行为；生产行为见第 4 点的 `*.zh-Hans.srt`。

## 2. 已实施的优化（B-3：歌词 LLM 重分段 + 翻译 prompt，Swift+C# 双端镜像）

`Translator.swift` / `Translator.cs` 的 `songLyrics` preset：

1. **segmentationGuidance（重分段层）**新增三点：
   - 主动把 Whisper 的碎段**合并成完整乐句**，不让助词悬空结尾、不把词拆两行；
   - **每行长度上限**（约 20 假名/汉字），过长乐句在内部自然停顿处再切（仍不拆词）——修复"合并过头导致一行 40+ 字、一屏放不下"；
   - **逐字保留每个假名/汉字、只可加标点、绝不改字**——直接降低 §1-F2 的对齐回退。
2. **translationGuidance（翻译层）**新增**具体 J-pop 重排范例**（「今日も渋谷の街に朝が」「降る…」→ 合成完整中文乐句；「怖くて仕方ない」「けど本当の自分」→ …），并强调不让某行以悬空成分结尾。anime preset 早有范例、songLyrics 此前没有，本次补齐。

**B-4 — 评测加牙齿（F5，已实施）**：`viewing_quality.py` 的 `weak_boundary_candidates` 新增 `danglingCaseParticle` 检测——cue 以宾格/主格助词 `を`/`が` 悬空结尾（高精度信号：这两个助词几乎不可能作一句字幕的自然结尾，必接后续谓语）。刻意不含 `に`/`の`/`は`/`も`（可合法收尾，纳入会误伤已断好的行）。`final_source_quality_issues` 在 music 类下把它计入 blocking。**实测**：真实群青 43 条源 → 命中 `weakBoundarySplit:3:danglingCaseParticle` 判 **FAIL**；B-3 重分段后的版本（无 を/が 悬空）→ **PASS**。给了"群青坏分段 fail、修好转 pass"的确定性回归门。测试 `tests/test_viewing_quality.py::test_final_source_flags_dangling_case_particle_and_clears_when_resegmented`。

**未做（含理由，遵循"最小正确改动/保护现有行为"）**：
- **源 planner（F1）**：实证否决，见上。
- **duplicate-noise 守卫收窄（原 B-2）**：分析表明其对群青精确重复副歌会被 `original != candidate` 跳过、很可能不触发；无证据支撑、且削弱真乱码防护，故跳过。
- **逐字对齐容错（LCS）**：高风险（可能错位时间轴），本机无 LLM 无法验证，列为后续；当前用 prompt 强化"逐字保留"规避。

## 5. 30 样本 Whisper 路线研究（跨 8 语言，Sub-agent 端到端模拟）

对 `songs30` 的 **30 个真实缓存 whisper 词级产物**（ja×16/en×4/zh×3/ko×3/yue/it/es/fr 各若干）跑真实确定性 ASR 分段（`moongate-cli local-asr-srt`），再用 30 个 sub-agent 各自模拟"重分段+翻译"（当前 prompt）并按字幕组标准打分。音乐是 Whisper 路线的代表性压力场景——**音乐平台字幕基本是垃圾、才会触发 Whisper；演讲类平台字幕通常可用、走不到 Whisper**。

**总均分 5.77；按语言：en 7.0 / it 8.0 / es 7.0 / ja 6.13 / fr 5.0 / ko 4.67 / zh 3.33 / yue 3.0。**

**结论 1：B-3 把分段问题解决了。** charAlign **30/30 全保留** → 生产严格对齐不会再回退（群青回退根因消除）；wordSplit 仅 2/30、danglingParticle 6/30、overLong 4/30。**群青本身现 8/10**（原始投诉点），lemon/cry-baby/billie/laura/yoru-ni-kakeru 同为 8。

**结论 2：新的天花板 = ASR 转写质量，按语言分层，且 prompt 救不了。** 主导问题是 `mistranslation`(35) + `repetitionGarbage`(24)，逐样本看几乎全由**源 Whisper 错字/乱码**驱动：
- **ja/en/it/es 干净转写 → 7-8 分**（接近字幕组）。
- **zh/yue → 3-4**：转写本身乱码（青花瓷「素胚勾勒」→「了出情话被风弄转」、浮夸→满篇近音字）。**zh/yue 源=目标，无翻译，用户直接看到乱码 ASR**。
- **ko → 4-5**：whisper 自信地听错（罗马/谐音混排）。置信度抓不住（BLACKPINK avg_prob 0.85 却是乱码）。
- 置信度部分可分：乱码的 jay-chou/eason avg_prob 0.73-0.75、低分词占比 23-24%，明显低于干净样本（0.90-0.95、2-6%）。

**已实施（B-3 续，Lever 1）**：songLyrics `translationGuidance` 的 ASR-错字处理从笼统改为**具体+保守**——针对"单看像词、入句却突兀矛盾"的孤立错字（青あざ→青朝、笑って→言って 范例），顺乐句意象译通顺、必要时弱化模糊，**但宁可含蓄笼统也绝不编造/放大**。Sub-agent A/B 验证（lemon「言って」、白日「メッセラ/ブレスやら」）：孤立错字被顺意象化解、乱码被丢弃保高置信意象、正确句不被误改写、无幻觉；代价是信息密度略降（保守偏置，方向正确）。Swift+C# 镜像。

**建议（未实施，需七月定）**：
- **本地 ASR 输出置信度信号**：对 zh/yue/ko 这类源 whisper 普遍低置信/乱码（avg_prob<~0.78 或低分词占比>~0.2）的视频，给"本地识别质量较低、字幕仅供参考"的 UX 提示——诚实告知而非呈现自信乱码。注意：whisper 是 fallback，**没有更好的源可换**，故这是 UX 而非质量提升；且对 ko 的自信误识无效。
- **非音乐题材未覆盖**：本轮只有音乐有缓存 whisper 数据。如要覆盖演讲/动漫对白/新闻等（测 anime/interview/default 等 preset），需七月提供这些视频或其 whisper 词级输出。
