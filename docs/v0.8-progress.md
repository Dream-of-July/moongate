- 2026-06-21: Emergency 4K/1080p regression fix tightened after the reported real-world failures. Root causes: visible top tiers previously used an unbounded `bv*+ba/b` selector, then an upper-bound-only `height<=2160` selector that could still resolve below 4K; HDR preference could rank lower-resolution HDR ahead of same-tier SDR; and subtitle burn-in previously defaulted to `maxBurnHeight = 1080`, so translated/burned outputs could silently become 1080p even after a 4K source selection. Swift and C# now generate exact-first selectors for every visible tier, e.g. `bv*[height=2160]+ba/b[height=2160]/bv*[height<=2160]+ba/b[height<=2160]`. HDR preference now interleaves each fallback branch as same-tier HDR, same-tier normal, then lower fallback, so 1080p HDR cannot outrank available 2160p SDR. macOS `StartDownload` also scopes stale `preferHDR` state to the selected format's `hdrAvailable`. ## Decisions

- Use whisper.cpp as the v0.8 local ASR runtime across macOS and Windows.
- Keep faster-whisper only for offline evaluation under `tools/subtitle_timing_eval`.
- Defer WhisperKit until after the whisper.cpp path is stable.
- Do not silently download ASR models.
- Do not silently download or install ASR runtimes; runtime adoption may only fill a path for an executable that already exists locally.
- Runtime bundling for release is opt-in from a locally staged, already-built whisper.cpp runtime directory. Build scripts must not fetch, clone, or install whisper.cpp during normal build, CI, or release packaging.
- Bundled or managed whisper.cpp runtime directories should include `asr-runtime-manifest.json`; when present, the app verifies platform, architecture, relative executable path, and SHA-256 before adopting the runtime.
- Build-time runtime packaging must write `asr-runtime-manifest.json` from the copied artifact, not from source-side metadata, so the hash covers the binary actually shipped in the app or Windows publish directory.
- Do not release, tag, upload assets, or publish v0.8 without explicit authorization.
- Normal CI should remain a quality gate, not an installation path: it must not download ASR models or bundled whisper.cpp runtimes.
- Numeric subtitle-continuation heuristics must mean decimal digits, not every Unicode numeric character; otherwise CJK place names such as `東京` can be misclassified.
- `TaskWorkPlan` is the forward path for local ASR progress. Old queue plans should only remain as compatibility input until download requests can create work plans directly from bytes, duration, ASR model/backend, subtitle count, and encoder history.
- Queue pause/resume is a state-change contract, not a fire-and-forget UI event. Pause must return failure when the task is missing, cancelled, or already paused, and queue capacity may only be released after the task control token accepts the pause. Parked resume must reacquire capacity before resuming a suspended process.
- ASR cache language matching treats `nil` and `auto` request language as auto-detect/wildcard. Cache entries should store the detected transcript language; explicit language requests remain strict against that detected language, so a cached Japanese transcript can satisfy a Japanese retry but not an English-forced rerun.
- UI-selected local ASR auto-detect uses the explicit track language code `auto`, but whisper.cpp command planning must omit `-l` for that value so the runtime performs language detection.
- Default local-ASR prompts must not mention `language=auto`; the `auto` value is an internal request sentinel, not a prompt hint for the recognizer.
- Local ASR source SRT files are durable intermediate artifacts. Queue retries must reuse an existing same-language `.local-asr.<lang>.srt` before invoking whisper.cpp again, especially when the previous failure happened in translation or burn-in rather than recognition.
- In the v0.8 ready page, "no subtitle source" and "save source subtitle only" are separate decisions. `ChineseSubtitleMode.off` is kept as the compatibility execution layer for saving the selected source without translation or burn-in, not as a user-visible no-subtitle state.
- Onboarding should not fork a second API settings implementation. The first-run non-local translation path reuses the shared API editor so model fetching, connection testing, and endpoint shaping match Settings.
- The queue local-ASR shortcut is a post-download enhancement path, not a replay of the original subtitle mode. It should reuse the downloaded video, create a local-ASR source, and default to translated burn-in so a plain completed download can become a subtitled video.
- Burn-in should preserve source resolution by default without letting low-bitrate sources balloon after re-encode. When source probe data is available, macOS and Windows use source/target short-side bitrate caps for both software and hardware burn-in paths; missing probe data falls back to the previous quality/CRF behavior.
- The Updates & About pane should not duplicate the app version in both sections. Keep version disclosure in the Update section and use About for app identity/source links.
- Compatible gateway model fetching should be forgiving about harmless request-shape differences. Use Anthropic pagination for official Anthropic, but let strict gateways retry the same endpoint without Moongate-added query parameters; keep OpenAI-compatible headers free of Anthropic-only fields.
- `/UPDATEPID` `OpenProcess` failure is treated as safe-to-continue because the app may already have exited; timeout and `WAIT_FAILED` after a valid handle remain abort conditions.
- ASR wire JSON should use path semantics for local files (`audioPath`, `transcriptPath`, `executablePath`). Swift decodes old `*Url` spellings for migration but no longer writes them.
- ASR recognizers should register long-running runtime processes through the queue control token where available. Pure cancellation tokens remain supported for tests and standalone calls.
- Audio extraction and whisper recognition must remain serial under one `TaskControlToken`; each process adapter must clear the active PID in `finally` / termination handling before the next process is registered.
- Real-runtime ASR QA must be an explicit release gate, not normal CI. It must require local runtime/model/audio/ffmpeg paths, must not download anything, and must validate JSON plus `*.local-asr.<lang>.srt` output rather than treating `whisper-cli` exit code alone as success.
- Manual local-ASR release QA should prefer the packaged `asr/runtime` directory from a built app or Windows publish output. The smoke must verify `asr-runtime-manifest.json`, ensure the executable path stays inside that directory, and compare SHA-256 before invoking whisper.cpp.
- The shipping app may instantiate the real local ASR generator only through the explicit settings factory. Missing, disabled, or invalid runtime/model/ffmpeg paths must produce a nil generator and keep the queue behavior non-destructive.
- Completion notifications should be emitted only when a batch reaches the all-done boundary, not once per task. App-layer status remains the first surface; system notification/toast support can improve later without changing the queue coalescing contract.
- Completion notification and sound defaults may be on, but the controls must be visible on both macOS and Windows so users can disable either surface without editing settings JSON.
- Windows v0.8 dark mode starts as system-following theme resources, not a new settings preference. User-facing theme selection can come later after the settings hub exists.
- Windows onboarding may let the user choose the future AI translation service provider, but it must not collect credentials. Provider selection only seeds default AI/translation protocol and base URL; API keys and model names remain explicit settings work after onboarding.
- macOS onboarding should preserve the Apple local translation option while also asking for the future API-compatible provider. Provider selection seeds default AI/translation service metadata for later setup; onboarding must not request credentials, runtime paths, model paths, or model downloads.
- Windows settings edits must preserve hidden settings by deriving from the original opened `AppSettings`; new hidden v0.8 fields should not require every visible settings page to remember to copy them manually.
- The Windows settings hub should reuse existing setting controls and save/cancel semantics while the information architecture settles. Deeper per-page rewrites, search, ASR model management, and notification/beta controls can layer on top of the tabbed shell.
- Windows settings page names should match the v0.8 target information architecture even while page content remains incremental. Components & Storage and Updates & About must contain visible storage/about content, not only renamed component/update pages.
- The macOS settings hub should mirror the v0.8 target pane names while reusing existing controls and save/cancel semantics. General stays lightweight; video performance controls belong with Video & Output, and Components & Storage / Updates & About must contain real storage/about content rather than title-only renames.
- A visible resolution tier must be an actual yt-dlp constraint, not just a label. 2160p/4K choices use exact-first selectors with bounded fallback, and subtitle burn-in preserves source resolution by default; 1080p burn-in scaling remains an explicit speed/size option.
- Resume guard: do not reopen the 4K/1080p regression merely because context was compacted. It is currently fixed and verified in the v0.8 working tree; only reopen it with new reproduction evidence, proof that the user is running a stale build, or a failing 4K regression test. If the context mentions this bug again without new evidence, treat it as prior-thread carryover, not a fresh user report.
- yt-dlp download percent is treated as current media-stream progress, not whole-task progress. For selectors that download video and audio separately, Moongate aggregates streams before updating user-visible queue progress, and it does not treat current-stream ETA as an exact whole-task ETA.
- Unknown queue ETA should say that Moongate is estimating time, not show fake precision. Chinese UI uses `正在估算时间` / `正在估算時間`; precise or approximate remaining time is shown only when the estimator has usable evidence.
- Expected media-download counts must parse yt-dlp selector syntax, not raw characters. Only top-level `+` means a video/audio merge; plus signs inside filter values such as `HDR10+` are metadata and must not affect progress/ETA accounting.
- Recommended v0.8 ASR model metadata must come from a trusted manifest, not live UI scraping. The current shipping candidate list is the quantized whisper.cpp `tiny-q5_1`, `base-q5_1`, and `small-q5_1` files from `ggerganov/whisper.cpp`; settings may install them only through an explicit user action that shows metadata first, downloads to staging, verifies SHA-256, and then moves the file into the managed model store.
- The real local-ASR generator factory treats recommended model IDs as verified managed models, not arbitrary file paths. For those IDs it must require a matching managed installed path and good SHA state; custom model IDs remain a manual advanced path.
- Local ASR model install failures are user-facing release errors, not developer enum states. Failure notices should name the model and the actionable cause, especially SHA-256 mismatch, missing downloaded file, or insufficient disk space.
- Storage cleanup may only operate on app-owned locations. Download history management must use `Downloads/Moongate` (or a future explicit app history index), never heuristics that scan and delete arbitrary folders from the user's whole Downloads directory.
- macOS settings UI should prefer native SwiftUI Form/List/SplitView structure, system symbols, clear action weight, and progressive disclosure. Local ASR setup should surface only readiness, recommendation, download/use/delete state, and concise capability text by default; executable/model path fields belong in Advanced.
- Update availability can be surfaced as lightweight badges, but update install/download behavior remains Sparkle-owned on macOS; sidebar badges should observe `UpdateService.updateAvailable` and must not trigger network or installer side effects.
- Release-candidate packaging may install ad-hoc signed local QA builds, but this is not a formal public release. Do not tag, publish GitHub assets, notarize, or claim signed production readiness without explicit approval and the missing signing credentials.
